OS-线程调度

Created2022-08-06|操作系统

LabWeek16 实验报告

设计思路

总体思路

  • 整个调度模拟器主要分为任务生成线程和任务调度线程两个线程来负责任务的生成和调度。任务生成线程负责初始化任务线程的参数并将其按一定的顺序加入到就绪队列中；任务调度线程负责根据给定的调度策略对就绪队列中的任务进行调度。

关于模拟

  • 利用sleep()函数睡眠一段时间模拟任务的工作时间。
  • 用一个全局变量isCpuBusy来进行表示CPU是否处于繁忙状态。
  • 调度的实现是通过每个任务结构体中的一个信号量sche，将其初始化为0，任务线程开始会使用sem_wait(&(task->sche));，而调度线程则是通过sem_post(&(cur->sche));来调度线程。若某个任务没有被调度，由于信号量初始为0，因此会阻塞在sem_wait(&(task->sche));，直到调度线程对其执行了sem_post(&(cur->sche));，该任务线程才可以正常执行。
  • 抢占的实现则是通过pthread_kill()函数通过对被抢占的任务线程发送信号，使其从sleep()函数中断，记录下sleep()函数返回的剩余时间，重新进入就绪队列以等待被重新调度。

三种策略

  • 实现FCFS只需要依次将输入的任务插到就绪队列的队尾即可，然后调度线程则是每当CPU处于空闲状态，就查看就绪队列中是否有就绪的任务，若有则将队首出队将其调度到CPU开始执行。
  • 实现SJB则需要维护一条以任务剩余时间为次序的队列，引入一个变量cur_shortest_time来记录当前执行任务的剩余时间，每当检测到当前队首的任务剩余时间少于当前执行的任务的剩余时间时，就会发生抢占。
  • 实现优先级调度则需要维护一条以优先级为次序的队列，引入一个变量cur_highest_priority来记录当前执行任务的优先级，每当检测到当前队首的优先级高于当前执行的优先级时，就会发生抢占。

具体实现

  • 整个程序使用的函数如下：
    • void* task_generator()：任务生成函数，是一个线程函数，负责读取键盘输入的任务参数，生成对应的任务插入到就绪队列中。
    • void* task_schedule()：任务调度函数，是一个线程函数，负责对就绪队列中的任务根据所选的策略进行调度。
    • void task_func(void args)：任务模拟函数，是一个线程函数，负责模拟任务的执行。
    • void insert_priority(task_info* p)：以优先级为排序标准将任务结点插入到就绪队列。
    • void insert_remaining_time(task_info* p)：以任务剩余时间为排序标准将任务结点插入到就绪队列。
    • void remove_node(task_info* cur)：将特定任务结点移出就绪队列。
    • void sighand(int signo)：信号处理函数。
    • void* myclock()：时钟函数，用于实时更新当前所执行的任务的剩余时间，以便判断能否抢占。
  • 使用的全局变量以及结构体如下：
  • void* task_generator()：任务生成函数
  • void* task_schedule()：任务调度函数
  • void task_func(void args)：任务模拟函数
  • void insert_priority(task_info* p)：按优先级插入函数
  • void insert_remaining_time(task_info* p)：按剩余时间插入函数
  • void remove_node(task_info* cur)：将结点移出队列
  • void sighand(int signo)：信号处理函数
  • void* myclock()：时钟函数

结果分析

输出格式

  • 首先，程序输出的格式如下：
    • 最上面是调度策略的选择，共有FCFS、SJB和普通的优先级调度三种，其中后两个均为抢占式调度。
    • 选择好调度策略之后（比如图中输入了1），后面的结果大致分为三列，最左边一列为输入，中间一列为任务的调度情况，最右边一列为任务线程情况。

FCFS

  • 首先测试等到当前任务执行完毕再输入新任务的情况。
  • 如上图，最开始输入1，此时便有了一个工作线程140328661153536，因此会调度它，由于是先来先服务的调度原则，因此调度之后会输出当前无任务可调度，最右边显示该线程开启，再过1秒，显示该线程结束。
  • 结束之后再输入第二个任务的wct——2，此时有了一个新的工作线程，同样地会对其进行调度，最右边显示该线程开启，过了2秒，显示该线程结束。
  • 再输入第三个任务的wct——3，后面的流程与上面相同，直到第三个任务线程结束。
  • 最后输入-1表示结束输入，由于每次是等到当前输入的任务完成之后再输入一个新的任务，故输出了平均等待时间为0秒。
  • 再测试连续输入任务的情况
  • 如上图，连续输入2个任务，wct分别为5和2，由于是FCFS，因此会先调度第一个任务，过了5秒之后，再调度第二个任务。
  • 最后结束输出了平均等待时间为2.5秒，与理论上的计算：（5 + 0）/2 = 2.5一致

SJB

  • 首先测试没有抢占的情况
  • 如上图，输入第一个任务的wct——5s，约过了1s之后，输入第二个任务，给定wct为10s，由于第二个任务此时的剩余时间长于第一个任务，因此不会发生抢占。
  • 可以看到2个任务线程被创建，但当第一个任务结束之后第二个任务才会开启。
  • 最后输入-1结束，输出的平均等待时间为2s，由于输入两个任务间隔时间约1s，故与理论计算：4 / 2 = 2一致。
  • 实际上，个人认为，由于wct是以秒为单位，且程序内部的等待时间最小精度均是秒，再加上调度可能有语句执行的时间延迟，因此最终结果误差在1到2秒之内都是可接受的。
  • 再测试有抢占的情况
  • 如上图，输入第一个任务的wct——10s，约过了1s之后，输入第二个任务，给定wct为5s，由于第二个任务此时的剩余时间为5s，长于第一个任务，因此会发生抢占。
  • 因此当第二个任务输入之后，会输出线程号为139825846732544的任务（即第二个任务）进行抢占，同时，会输出线程139825855125248（即第一个任务）被抢占。
  • 过了5秒之后，第二个任务结束，此时第一个任务重新开启，此时剩余时间为9秒，过了9秒之后，第一个任务结束。
  • 最后输入-1结束，此时平均等待时间为2.5s，与理论计算5 / 2 = 2.5一致。

优先级调度

  • 首先测试没有抢占的情况
  • 如上图，由于优先级1~10，1最高，10最低，先输入第一个任务，优先级为1，wct为3，约过了1秒再输入第二个任务，优先级为2，wct为1。
  • 由于第二个任务优先级低于第一个任务，因此不会发生抢占。因此当第一个任务结束之后，才会开启第二个任务。
  • 最后结束输出平均等待时间为1.5s，与（3 - 1）/ 2 = 1一致。
  • 再测试有抢占的情况
  • 如上图，先输入第一个任务，优先级为2，wct为10，再紧接着输入第二个任务，优先级为1，wct为2。
  • 由于第二个任务优先级高于第一个任务，因此发生抢占。可以看到，第一个任务被抢占，当前的优先级由2变为1。过了2秒，第二个任务结束之后，第一个任务才重新开启，当前优先级重新变为2。
  • 最后结束输出平均等待时间为1s，与理论计算：2 / 2 = 1一致。
