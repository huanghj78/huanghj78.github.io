Algorithm 6-1

实验原理

  • 此程序主要是验证fork函数的相关使用。一个进程可以通过调用fork()函数尝试产生一个新的进程。在此程序中，通过在主函数中先定义了一个变量count，然后分别在两个进程中修改查看对比count的值以及地址来探索调用fork()函数进程的产生机制。再通过观察程序语句的执行来探索父子进程之间的运行关系。

实现细节

  • 头文件作用
    • : Unix/Linux系统的基本系统数据类型的头文件，含有size_t，time_t，pid_t等类型。程序中声明了一个pid_t型的变量childpid用于存储进程号。
    • : Linux/Unix系统中内置头文件，包含了许多系统服务的函数原型，例如read函数、write函数和getpid函数等。其作用相当于windows操作系统的”windows.h”，是操作系统为用户提供的统一API接口，方便调用系统提供的一些服务。
    • : 用于调用函数wait()
  • 首先，声明了一个pid_t型的变量childpid用于存储进程号。
  • 调用函数fork()将其返回值赋值给childpid，若fork()创建新进程失败，则会返回一个小于0的数，若成功则会赋给子进程中的childpid一个0而赋给父进程的childpid一个大于0的数。因此可以通过一个条件分支语句分别定义父进程和子进程的不同操作，故父子进程中childpid值的不同是实现进程异步执行的关键。

  • 故接下来根据childpid值的不同定义不同的操作，若判断childpid小于0，则执行语句

    输出fork()函数出错的原因并退出程序。
    若不小于0则证明成功创建了一个新的进程。

  • 由于fork()函数创建进程的机制是拷贝一份父进程的PVM给子进程，故实际上相当于有两个相同的程序同时在运行，因此就可以通过在代码中利用条件判断childpid的值使得两个程序执行不同的分支，从而实现了两个进程的异步执行。
  • 父进程执行的操作：父进程会执行childpid > 0对应的程序块，即输出父进程的进程号，子进程号，count的值以及count的地址，然后调用函数sleep(5)睡眠5s后调用函数wait(0)等待子进程的结束，最后输出一条输出语句后结束进程。
  • 子进程执行的操作：子进程会执行childpid == 0对应的程序块，即将count自增，并输出子进程的进程号，count的值以及count的地址，然后执行一条输出语句后结束进程。

结果分析

  • 从前两条输出可看出，父进程的进程号为21328，子进程的进程号为21329。
  • 父进程输出的count的值为1，而子进程输出为2，从这一点可以看出两个进程中的count实际上是不同的两个变量，即，两个不同的物理地址，由于子进程的PVM是由父进程的PVM拷贝过来，因此栈结构是相同的，这就是两个进程输出的count的地址（虚拟地址）是相同的。
  • 后两条输出的语句依次为子进程和父进程所输出。因为父子进程执行完不同的内容之后都会执行最后一条输出语句，而其顺序是一定的，原因在于父进程最后调用了函数wait(0)，只有等到其所有的子进程都结束了自身才能够结束。（若该父进程只有这一个子进程，也可以将wait的参数改为子进程的ID，即等待特定的子进程结束。

代码修改和分析


  • 在父子进程分别定义变量test，父进程赋值为2，子进程赋值为1，从输出可看到两个test的值是不同的，而虚拟地址相同，进一步可以证明子进程得到一个独立的PVM，但该PVM的布局与父进程相同。

Algorithm 6-2

实验原理

  • 此程序主要是验证vfork()函数的相关机制。vfork()与fork()最大的不同在于vfork并没有给子进程提供一个新拷贝出来的PVM，而是将父进程原有的PVM先让给子进程使用，当子进程使用完毕结束进程之后父进程才能恢复正常。因此可知父子进程的资源是共享的，且无法并发执行，而其优点也明显，即节省了资源。

实现细节

  • 一开始与第一个程序一样，根据调用vfork()函数的返回值定义不同的操作。
  • 在子进程部分，先让count自增，然后输出进程号，count的值及地址，然后调用sleep(10)函数睡眠10s，最后调用函数_exit(0)结束进程。需要注意的是，在这里应该通过_exit(0)函数（内核态）或exec(0)函数（用户态）来结束子进程，而不能使用return语句，原因在于父子进程共享同一个PVM，若在子进程调用了return语句，则会对该PVM做栈操作，导致父进程的资源丢失或混乱。
  • 在父进程部分，先输出父子进程的进程号，count的值以及地址，然后调用函数wait(0)。需要注意的是，虽然单纯看此程序，父进程肯定是在子进程结束后才恢复执行，并不需要等待该子进程，但有可能该父进程还有其他的子进程，因此需要养成在父进程最后调用wait()函数的编程习惯。
  • 在代码的最后是一句输出语句以及return语句结束父进程。

结果分析

  • 父进程号为26125，子进程号为26126
  • 从输出语句的顺序可知，当子进程在执行的时候，父进程并不会执行。父子进程中的count实际上的物理地址是相同的，尽管子进程和父进程输出的count值相等并不能完全说明这一点。
  • 当子进程结束之后，父进程才执行其输出语句，程序最后的输出语句只输出了一次，这与第一个程序不同，原因在于子进程在子进程代码模块的最后就通过调用_exit(0)函数而结束进程，故不会执行到父进程代码块后面的代码。

实验原理

  • 此程序在第一个程序的基础上做了修改，主要是验证父进程没有调用wait()函数的结果。

实现细节

  • 在子进程模块中，在第一个程序原有的基础上调用了函数sleep(10)使得子进程的执行时间比父进程长，而在父进程中只有一句输出语句，没有wait()函数。

结果分析

  • 父进程号为29830，子进程号为29831
  • 从输出的结果可看出，父进程执行完输出语句，直接执行程序的最后一个输出语句后结束进程。而子进程执行完其输出语句后进入了睡眠。
  • 在子进程睡眠的时间段，在终端输入命令ps -l查看进程可发现父进程已经不在，子进程变为了孤儿进程。子进程的原本的PPID应为29830，但现在变为了856，可见其已被系统进程所收留。
  • 子进程结束睡眠之后同样执行了程序的最后一个输出语句而结束进程，但因为其原本的父进程早已结束，最后的终端界面出现了异常，按回车键即可恢复正常。

Algorithm 6-4

实验原理

此程序在程序3的基础上，在父进程的最开始便调用了wait()函数，此时父进程会停止工作等待子进程的结束之后才能继续工作。

实现细节

  • 子进程通过调用函数sleep(5)使得其比父进程原本的执行时间长
  • 父进程在最开始便调用了函数wait()并把其返回值传给pid_t型的变量terminatedid，然后在wait()的后面是一个输出语句。

结果分析

  • 父进程号为15443，子进程号为15444
  • 从输出结果可看出，当子进程在执行的时候，父进程并不会执行，原因在于父进程在最开始便调用了wait()函数而进入等待状态。由于参数为0，故只有当所有子进程结束之后父进程才能继续执行。故可看到输出结果最开始均是子进程的输出。
  • 当子进程结束的时候，父进程开始输出，可以看到输出的terminated pid = 15444，即可看到父进程所在等待的子进程的ID是多少。
  • 由于是使用fork()函数创建进程，故与第一个程序一样，父子进程输出的count是不同的。

Algorithm 6-5

实验原理

sleeper.c

  • 首先，写了一个sleeper的程序用于替代父进程生成的子进程.sleeper程序的主要功能就是调用sleep()函数睡眠一段时间以表示一个进程占用一段时间。

vfork-execv-wait.c

  • 此程序演示了如何通过vfork()和execv()函数的搭配使用来调用开启新的进程，同时展示了wait()函数在此程序中的功能。
  • 通过execv()函数，弥补了vfork()函数的缺点，即两个进程无法并发执行的问题，同时又发挥了vfork()的优势，因为若使用fork()来创建一个新进程，新进程的PVM是完全拷贝父进程的PVM，这个过程需要花费较多的时间，而且可能父进程许多原有的资源并不会在子进程中用到，所以意义不大；而vfork()并不会拷贝出新的PVM，因此可在调用vfork()生成的子进程中调用execv()函数来开启一个新进程，而此进程会生成一个新的PVM，从而提高进程赋值效率。

实现细节

  • 头文件作用
    • : unix/linux系统定义文件状态所在的伪标准头文件。
  • 首先调用vfork()函数生成一个子进程，这个子进程执行的工作为：先输出相关信息，然后睡眠2s.
  • 然后定义一个字符数组用于存放新进程的文件名和一个stat类型的结构体变量buf，调用stat函数获取alg.6-5-sleeper.o文件的信息，然后通过execv()函数运行事先编译好的文件alg.6-5-0-sleeper.o，从而开启一个新的进程，调用之后原有的子进程则会自动结束并把其PID转交给execv()函数开启的进程。
  • execv()函数开启的进程拥有自己独立的PVM，而原本占用父进程PVM的子进程已经结束，因此此时父进程可以继续执行，从而实现了两个进程之间的异步执行。
  • 需要注意的是，一般在子进程创建之后不要再有其它的系统调用而是直接调用execv()函数转到一个外部可执行文件开启新进程，若在此之前还有其它系统调用，如printf()等，可能会导致出错。

结果分析

  • 父进程号为52187，子进程号为52188
  • 从输出结果可以看到子进程在睡眠的时候父进程并不会执行
  • 开启sleeper进程之后，父进程开始执行，输出信息，同时sleeper进程也在执行，输出其信息，可以看到它的PID为52188，即为原来的子进程，而它的PPID为52187，即为父进程。然后sleeper进程进入了睡眠，进入睡眠之后父进程并不会就此结束，因为在其中调用了wait()函数，故等待作为其子进程的sleeper进程的结束。
  • 当sleeper进程结束睡眠并结束进程之后，父进程继续执行，输出了wait()函数的返回值——15477，即sleeper进程的PID，可以得知父进程确实是在等待sleeper进程的结束。

Algorithm 6-6

实验原理

  • 此程序在第五个程序的基础之上，在父进程代码模块进行了修改，取消了wait()函数，同时调用系统函数system(“ps -l”)，在终端显示进程信息，从而演示了当使用vfork()函数和execv()函数而不使用wait()的结果。

实现细节

  • 父进程模块先输出进程相关信息，然后调用系统函数system(“ps -l”)，然后调用函数sleep(1)以达到同步。其它模块与第五个程序相同。

结果分析

  • 父进程号为59162，子进程号为59163
  • 从输出结果可看到，子进程在睡眠的时候父进程并不会执行，而当子进程开启sleeper进程后结束自身之后，父进程开始执行。
  • sleeper进程进入睡眠的同时，父进程输出其进程相关信息，然后通过用系统函数system(“ps -l”)，显示进程信息，可以看到bash是”父进程”的父进程，而sleeper进程取代了原来的子进程，成为了父进程的子进程，而调用的系统函数system()所开启的进程sh同样也是父进程的一个子进程。执行指令”ps -l”之后生成的进程ps则是进程sh的子进程。
  • 在父进程结束之后，在终端手动输入指令”ps -l”查看进程信息可发现，父进程已经消失，而sleeper的PPID由59162变为856，即sleeper已被856号进程收留。
  • 通过指令”ps -q 856”，查询856号进程的信息可知其为systemd的守护进程
  • 由于父进程中没有调用wait()函数，故父进程结束之后sleeper进程仍在睡眠，故最后bash进程和sleeper进程是异步执行。

此程序代码没有进行修改。
