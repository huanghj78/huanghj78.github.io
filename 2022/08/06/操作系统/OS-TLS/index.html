alg.14-1

原理性解释

  • 首先，前面的程序（从1到5）是关于TLS的相关内容。
  • TLS，即线程局部存储，是指实现一个在线程内部的各个函数调用都可以访问，但其它线程无法访问的变量的机制。
  • 此程序是通过编译级别声明**__thread**来实现TLS。
  • __thread是GCC内置的线程局部存储设施，对于**__thread**变量，每个线程有一份独立实体，各个线程的值互不干扰。

细节性解释

  • 头文件<sys/syscall.h>：提供了一些系统调用
  • define gettid() syscall(__NR_gettid)封装了一个系统调用 __NR_gettid
  • 定义了一个TLS类型的整型变量tlsvar，用__thread修饰提示编译器对其进行处理。
  • 定义线程函数thread_worker其实现的是随机sleep()一段时间（for(int k = 0;k < randomcount;k++);的作用），然后输出其局部变量tlsvar并使其自增，循环五次。
  • 接着在主函数就是通过调用两次pthread_create()函数创建两个子线程执行上面的线程函数，然后主线程也实现同样的操作。
  • 此程序就是为了验证TLS的作用，即，在各个线程中，tlsvar的值是独立互不影响的。

结果分析

  • 可以看到，三个线程异步地执行，它们输出的顺序是随机的，但它们的tlsvar的值都是从0依次递增到4，彼此之间互不影响，就像是该线程内的一个局部变量，可见__thread实现TLS的有效性。

alg.14-2

原理性解释

  • 此程序是使用另外一种方法实现TLS，即通过POSIX的一些函数，具体如下
    • int pthread_key_create(pthread_key_t* key, void (*destr_function) (void *))
      • 功能是用于创建一个pthread_key_t类型的变量，将值传给函数的第一个参数（故需传地址）。
      • 第二个参数为一个清理函数，在线程释放该线程存储的时候被调用。该函数指针可以设成 NULL ，这样系统将调用默认的清理函数。
    • int pthread_setspecific(pthread_key_t key, const void *value)
      • 功能：将由pthread_key_create函数生成的key与第二个二参数，即想要作为tls变量的变量进行绑定
      • key：需要关联的键
      • value：指向需要关联的数据
      • 返回值：成功返回0.其他任何返回值都表示出了错误
    • void *pthread_getspecific(pthread_key_t key)
      • 功能：获取调用线程的键绑定，并返回其地址
    • int pthread_key_delete(pthread_key_t key)
      • 功能：销毁线程特定数据键
      • key：需要删除的键
      • 返回值：成功返回0.其他任何返回值都表示出了错误
  • 此程序要实现的是通过在每个子线程输出tls变量的地址以及利用tls变量打开一个文件并写入信息来验证POSIX实现TLS的有效性

细节性解释

  • 首先声明一个静态全局的pthread_key_t类型的变量log_key用于后面绑定tls变量。
  • 然后定义一个子线程将会执行的一个函数write_log()用于根据所在线程的tls变量打开对应的文件并写入特定信息。具体的实现是定义一个文件指针fp_log用于接收调用pthread_getspecific()返回的与在子线程中通过pthread_setpecific()与log_key绑定的tls变量，然后写入文件并输出该变量的信息。
  • 线程函数中实现的则是同样定义一个文件指针fp_log，然后用其调用fopen()打开一个特定的文件，接着调用pthread_setpecific()将fp_log与log_key绑定，然后调用write_log()
  • 接着定义一个类似于类的析构函数功能的一个函数，作为调用pthread_key_create()函数的清理函数，其实现的功能是关闭文件。
  • 最后主函数中实现的则是创建5个子线程，然后在终端执行lsof +d ./log用于列出log目录下的打开状态的文件，cat ./log/thread-1.log ./log/thread-5.log用于显示thread-1.log和thread-5.log文件中的内容。

结果分析

  • 需先建立一个名为log的文件夹
  • 可以看到，五个子线程分别输出tls变量的信息，对应的键值log_key均为0，但fp_log的地址各不相同，可见是五个不同的局部变量。
  • 执行lsof +d ./log之后没有显示，因为都执行了清理函数被关闭。
  • 执行cat ./log/thread-1.log ./log/thread-5.log之后可以看到两个文件中对应的内容。
  • 若将pthread_key_create()第二个参数改为NULL，即调用默认的清理函数，结果如下
  • 可以看到五个被打开的文件
  • 从结果可以验证各个函数的功能，以及POSIX实现TLS的有效性。

alg.14-5

原理性解释

  • 此程序与前面程序的不同在于tls_key的绑定不在子线程中，而是在子线程所调用的函数中。然后通过在一个子线程分别调用两个函数的对比————一个函数是将tls变量声明在栈上，另一个函数是将tls变量是声明在堆上。

细节性解释

  • 主函数完成的则是创建tls键值，创建线程等。
  • 线程函数完成的是依次调用thread_data1()和thread_data2()并输出对应结构体中的内容，最后释放thread_data2()函数申请的内存空间。
  • thread_data1()函数通过sturct msg_struct ptr[5]在栈上声明了一个结构体数组，然后对其内容进行负责，最后返回。
  • thread_data2()函数则是通过ptr = (struct msg_struct*)malloc(5*sizeof(struct msg_struct))在堆上声明了结构体数组，并同样对其内容进行负责，最后返回。

结果分析

  • 可以看到，在子线程的调用的函数中绑定tls_key是有效的，两个结构体数组对应了不同的地址，但是由于thread_data1()函数中是将结构体数组存放在其栈上，当函数退出之后就被销毁，因此里面的内容已经无效，而存放在堆上的只要不释放就仍有效。

alg.14-6

原理性解释

  • 此程序是关于clone()函数的使用，关于clone()的信息如下
  • int clone(int (*fn)(void *), void *child_stack, int flags, void *arg);
    • 其功能是完成一个任务（task）的创建，根据其传入的参数的不同可视为进程或线程。
    • fn：函数指针，即创建出来的任务要执行的函数。
    • child_stack：为创建的任务分配的堆栈空间
    • flags：标志位，指定了创建出来的任务与父任务之间的关系
    • args：传给fn函数的参数
  • flags主要的标志位有：
    • CLONE_PARENT：创建的子进程的父进程是调用者的父进程，新进程与创建它的进程成了“兄弟”而不是“父子”
    • CLONE_VM：子进程与父进程运行于相同的内存空间
    • CLONE_VFORK：父进程被挂起，直至子进程释放虚拟内存资源
    • CLONE_FILES：调用进程和子进程会共享相同的文件描述符表
    • CLONE_SIGHAND：子进程与父进程共享相同的信号处理（signal handler）表
    • CLONE_NEWIPC：创建的进程的IPC处于一个新的IPC命名空间
    • CLONE_THREAD：子进程与父进程共享相同的线程群
  • 此程序主要用到的标志位是CLONE_VM和CLONE_VFORK，另外还用到了一个标志位SIGCHLD，使得父进程可以接收子进程的信号

细节性解释

  • 首先定义进程函数child_func1()和child_func2()，它们实现的是将传入的字符串中的tid和pid修改为自身的值并输出，然后sleep一段时间后退出。
  • 主函数实现的是通过malloc为两个进程申请一个内存空间作为栈。
  • 然后在定义的字符数组中写入主函数的pid，sleep一秒之后，利用执行主函数时传入的参数作为flags调用两次clone()函数分别执行child_func1()和child_func1()。
  • 最后通过wait(-1,&status,0)等待任意一个子进程退出，等到之后sleep一秒，然后在终端执行“ps”，最后释放申请的内存空间之后退出。

比较pthread和clone()线程实现机制的异同

  • 首先，pthread()是POSIX提供的API，因此它是基于用户线程实现，而clone()是系统调用，因此是基于轻量级进程来实现。
    • 对于用户级线程，内核是无法感知多线程的存在的，内核仍是以进程为单位给该进程一个执行状态，因此当其中有一个线程阻塞，则整个进程都会阻塞。
    • 而轻量级进程是由内核支持的，每个轻量级进程是一个独立的调度单元，因此当有一个阻塞时，不会影响整个进程的执行。

相同点

  • 使用clone()时，若没有设置CLONE_FS等标志位，创建的轻量级进程实际上类似于fork()创建的进程，但设置了相关的标志位之后则类似于pthread_create(),也就是说，clone()的作用效果更宽泛，pthread_create()可以看作是clone()加上一些标志位的作用结果。

不同点

  • 由于pthread_create()，是通过用户线程的线程库pthread来创建用户线程，由线程库调度，因此内核是感知不到用户线程的存在的。由于其是用户级线程，因此是可以跨操作系统运行的，移植性较好。同时，由于其运行在用户层，不需要内核模式就可实现线程的切换，从而节省了系统开销。但对于多线程，当有一个线程阻塞，整个进程也随之阻塞。
  • clone()创建的是轻量级进程，内核是可以感知的，受内核的控制，因此会造成比较大的系统开销。但是如果创建出来的轻量级进程出现了阻塞，是不会影响整个进程的执行的。

程序分析：

  • 此程序首先通过在父子进程分别输出对应的PID和PPID来验证是否处于同一个线程群，进而验证CLONE_THREAD的作用。
  • 再者，由于使用CLONE_THREAD之后，当其中一个线程调用execv()函数执行一个新的程序之后，线程群中的其他线程会被终止，因此此程序也通过在一个线程中调用execv("./hello.o", buf);观察其他线程是否会终止来进一步验证CLONE_THREAD
    运行结果：
  • 上图为配置了CLONE_THREAD的结果
  • 可以看到，主进程和由其通过clone()创建的新进程的PID和PPID都是相同的，即他们处于同一个线程组中，而它们的TID是不同的，因为TID是区分一个线程组中各个线程的标志。因此，当使用CLONE_THREAD时，类似与线程的创建，而不使用CLONE_THREAD时类似于进程的创建。
  • 同时，可以看到，由于在child process1中通过execv()执行了hello.c，因此主函数会被终止，故看不到child process2的执行以及主函数的退出。
  • 上图为将CLONE_THREAD注释掉的结果
  • 当没有CLONE_THREAD时，创建的两个新的进程与主进程的PID各不相同，且两个新的进程的PPID与主进程的PID相同。
  • 同时，当child process1执行hello.c之后，主进程不会被终止，因此主进程可以创建child process2，故最后可以看到child process2和主进程的结束。
